Ingest Stage — Module Specification

Smart Ingestion Pipeline

interaction receipt → enrichment → issue linking → Customer Impact Discovery handoff

Phase 1
Intake & Gating
receive interaction → quality gates → bot detection → channel routing
Gate Processing Gate — Voice: <2 parties OR ≤60s → skip. Text: <2 parties OR ≤200 chars → skip.
Gate Bot-only exchange? — Checks participant metadata. If yes → org config: process_auto_agent_interactions? (default: false → skip & log bot_only_exchange)
Skip Failed gates → store skip_reason, exit pipeline
Route Channel Detection — routes to Voice, Chat, or Email processing path
↓ Channel-specific processing
Phase 2
Content Processing
transcription / text cleaning → sentiment derivation → normalisation
Voice Path
Run transcription engine
Audio formatting: language detection, punctuation restoration, speaker diarisation
Derive sentiment labels
Text Path (Chat / Email)
Strip & clean: headers, disclaimers, signatures, bot phrases, emojis
Derive sentiment labels
Signal Sentiment labels — start_sentiment_label + end_sentiment_label (negative | neutral | positive | null). Text classification, not full LLM. Feeds: positive_sentiment_shift (§4.4) and sentiment_trajectory_multiplier (AA §11.2).
Clean Field cleansing & normalisation — channel-agnostic from here. Flatten nested CRM/CCaaS metadata. Standardise timestamps, field labels, durations.
↓ Paths converge — normalised content
Phase 3
Enrichment & Customer State
metadata injection → customer profile lookup → stream & promotion resolution
Meta Metadata injection — interaction_id, agent_id, queue, channel, timestamps
Class Classify handling_mode & bot_participation_flag
Lookup Resolve customer_id — match inbound identifiers to Customer Profile Object
✓ Profile Found
Load existing Customer Profile Object
Proceed to state retrieval below
⚑ No Profile Found
Create new Customer Profile Object
Initialise with available metadata (identifiers, channel, timestamps)
Financial fields empty until CRM enrichment or first scoring pass
New customers · first-time contacts · unmatched identifiers
Retrieve Customer state retrieval — pull from Customer Profile Object:
Interaction History
repeat_contact_count, activity within context_window
Financial Snapshot
CLTV, cltv_confidence, cost_to_serve
Open Issues
Customer + churn issues, status, categories, reason texts
Linked Streams
Previously assigned stream_id(s) from prior interactions — informs stream resolution
Resolve Stream & Promotion Resolution — three-tier: metadata match → mapping rule → LLM inference. Set confidence scores & assignment source.
↓ Enriched interaction + customer state
Phase 4
Issue Linking & Handoff
customer issue linking (Stage 1) → BEI snapshot → payload build → Customer Impact Discovery handoff
Link Customer Issue Linking (Stage 1) — hybrid: rule-based + LLM tiebreaker. Uses contact_subcategory, repeat_contact_subcategory, reason text, open issue list. Customer issues only — churn linking happens in Customer Impact Discovery (Stage 2).
✓ Match Found
Confidence ≥ threshold
Link to existing open customer issue → set issue_id
+ New Issue
No match / new topic
Create Issue Object → set issue_id
⚑ Flag for Review
Below confidence threshold
Create new + flag
"over-create > mis-link"
Snapshot Capture bei_previous from Customer Profile — frozen before downstream processing. Critical for delta computation in revenue attribution.
Build Build normalised payload — enriched interaction + issue_id (nullable) + bei_previous
Store Store to validated storage + emit downstream flags
Handoff Route to Customer Impact Discovery — payload includes: normalised transcript, sentiment labels, customer state, issue_id, bei_previous, stream config
