Stage: Ingest

Smart Ingestion Pipeline

interaction receipt → enrichment → customer state → handoff

Phase 1
Intake & Gating
receive interaction → quality gates → bot detection → channel routing
Gate Processing Gate — Voice: <2 parties OR ≤60s → skip. Text: <2 parties OR ≤200 chars → skip.
Gate Bot-only exchange? — Checks participant metadata. If yes → org config: process_auto_agent_interactions? (default: false → skip & log bot_only_exchange)
Skip Failed gates → store skip_reason, exit pipeline
Route Channel Detection — routes to Voice, Chat, or Email processing path
↓ Channel-specific processing
Phase 2
Content Processing
transcription / text cleaning → sentiment → classification → normalisation
Voice Path
Run transcription engine
Audio formatting: language detection, punctuation restoration, speaker diarisation
Text Path (Chat / Email)
Strip & clean: headers, disclaimers, signatures, bot phrases, emojis
Signal Sentiment labels — start_sentiment_label + end_sentiment_label (negative | neutral | positive | null). Text classification, not full LLM. Feeds: positive_sentiment_shift (§4.4) and sentiment_trajectory_multiplier (AA §11.2).
Class Classify handling_mode & bot_participation_flag
Normalise Customer metadata normalisation — stitch identity and context across CCaaS/recording, CRM, ticketing, digital events into unified customer metadata. Provides clean identifiers for customer_id resolution and clean product metadata for stream matching.
↓ Normalised content + customer metadata
Phase 3
Customer State & Stream Attribution
stream resolution (Tiers 1–2) → customer profile lookup → state retrieval
Resolve Stream Attribution (Tiers 1–2) — attempt deterministic stream resolution from normalised interaction metadata:
Smart Ingestion — Deterministic Resolution
Tier 1: Metadata Match
Normalised interaction metadata explicitly identifies the product/stream
e.g., CRM metadata says "credit card product"
Tier 2: Mapping Rules
No explicit metadata — apply configured rules
e.g., if queue_id = credit_card → Credit Card Stream
✓ Resolved → set stream_id, confidence score & assignment source. Compute resolved_sltv (A × B × C or flat default) — Streams & Promotions resolution complete.
⚑ Unresolved → stream_id null, resolved_sltv null. Deferred to Customer Impact Discovery (Tiers 3–4) where resolved_sltv is computed once stream resolves.
Lookup Resolve customer_id — match normalised identifiers to Customer Profile Object
✓ Profile Found
Load existing Customer Profile Object
Retrieve state across all six domains below
⚑ No Profile Found
Create new Customer Profile Object
Initialise with available metadata (identifiers, channel, timestamps)
State, scoring, financial, issues, history all empty/default
New customers · first-time contacts · unmatched identifiers
Retrieve Customer state retrieval — pull from Customer Profile Object:
Identity
Identifiers, metadata normalised from CCaaS/CRM/ticketing
Streams
Linked streams, promotions, stream-level value
State & Scoring
Friction, risk modifiers, BEI, churn intent
Financial Data
Lifetime value, cost to serve
Issues
Customer issues, churn issues
Interaction History
Prior interactions, repeat contact, recent activity
↓ Customer state assembled
Phase 4
Handoff
snapshot → payload build → Customer Impact Discovery handoff
Snapshot Freeze entire customer state payload — current values locked until full processing is complete. bei_previous explicitly preserved for delta computation in revenue attribution.
Build Build normalised payload — normalised transcript, metadata, sentiment labels, frozen customer state snapshot including financial and scoring state, linked_streams, open_issues, interactions_in_context_window, stream config (nullable if unresolved), resolved_sltv (nullable if stream deferred to CID — computed there once Tiers 3–4 resolve)
Store Store to validated storage + emit downstream flags
Handoff Route to Customer Impact Discovery
Customer Impact Discovery — Inferential Resolution (Tiers 3–4)
When Tiers 1–2 cannot resolve — "we don't have the instructions, how should we classify this ourselves"
Tier 3: LLM Inference
No metadata, no rules — LLM reads conversation context to infer stream
Tier 4: Linked Streams
Fuzzy resolution using open issues and repeat contact within context_window against customer profile linked streams
↳ Tiers 3–4 execute in Customer Impact Discovery. Once stream resolves, resolved_sltv is computed there — Streams & Promotions resolution complete at that point.
